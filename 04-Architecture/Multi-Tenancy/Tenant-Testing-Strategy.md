## Purpose

This document defines the mandatory testing strategy for tenant isolation across the Kairo platform. It specifies what must be tested, how tests are structured, and what failures block release.

Tenant isolation cannot be proven through architecture documentation alone. It must be validated through tests that actively attempt to breach isolation and verify that the platform denies every attempt. Tests are the evidence that isolation works under real conditions.

---

## Scope

This document covers:

- Testing requirements for every tenant-aware surface.
- Test categories and their purpose.
- The reusable negative-test pattern.
- Module onboarding requirements.
- Release gate definition.
- Testing maturity model.

This document does not cover:

- Framework-specific test code or syntax.
- Functional test requirements unrelated to tenant isolation.
- Performance testing methodology (except noisy-neighbor).
- Test environment provisioning procedures.

---

## Foundational Principles

- **Happy-path tests are insufficient.** A test that verifies "Tenant A can read Tenant A's data" does not prove isolation. Isolation is proven by verifying "Tenant A cannot read Tenant B's data."
- **Tests must use at least two tenants where isolation is relevant.** Single-tenant tests cannot verify isolation. Every isolation test operates with multiple concurrent tenants.
- **Predictable or valid resource IDs must not bypass authorization.** Tests must verify that knowing a resource ID from another tenant does not grant access. UUIDs are not security.
- **Tenant-isolation failures block release.** A failing isolation test is a Critical-severity issue. It prevents deployment until resolved.
- **AI-generated implementation must include tenant-isolation tests.** Code generated by AI agents follows the same testing requirements as human-written code. No exceptions.
- **Test fixtures must not accidentally share ownership.** Test setup must create independent tenants with independent data. Shared fixtures create false-positive isolation tests.

## Testing Requirements by Surface

### 1. Tenant Resolution

| Requirement | Test Must Verify |
|-------------|-----------------|
| Context resolved from credentials | A request with Tenant A's credentials resolves to Tenant A's context |
| Client-supplied tenant ID does not override | A request claiming Tenant B in a header while authenticated as Tenant A resolves to Tenant A |
| Missing credentials fail | An unauthenticated request is denied (401) |
| Invalid credentials fail | A request with an invalid/revoked token is denied (401) |

### 2. Organization Membership

| Requirement | Test Must Verify |
|-------------|-----------------|
| Active member has access | A user who is a member of Org A can access Org A |
| Non-member is denied | A user who is not a member of Org B cannot access Org B |
| Suspended member is denied | A suspended user in Org A is denied access to Org A |
| Membership removal is immediate | After removal, the user cannot access the org on the next request |

### 3. Store Access

| Requirement | Test Must Verify |
|-------------|-----------------|
| Assigned store is accessible | A user assigned to Store 1 can access Store 1 |
| Unassigned store is denied | A user assigned only to Store 1 cannot access Store 2 |
| Org-level user accesses all stores | An org-admin can access any store in the org |
| Cross-org store access is denied | A user in Org A cannot access stores in Org B |

### 4. Resource Ownership

| Requirement | Test Must Verify |
|-------------|-----------------|
| Owned resource is accessible | Tenant A can read resources owned by Tenant A |
| Other tenant's resource returns 404 | Tenant A requesting a resource owned by Tenant B receives 404 (not 403) |
| Valid ID from another tenant does not grant access | Using a known resource ID from Tenant B does not expose the resource to Tenant A |
| Resource creation assigns correct ownership | A resource created by Tenant A has Tenant A's organization ID |

### 5. Role and Permission Scope

| Requirement | Test Must Verify |
|-------------|-----------------|
| Permissions apply only within the granted org | A user with `catalog.product.read` in Org A cannot read products in Org B |
| Store-scoped permissions are bounded | A Store Manager for Store 1 cannot manage Store 2 |
| Role in one org has no effect in another | An admin in Org A has zero permissions in Org B |
| Deny-by-default | A user with no assigned role has no access |

### 6. Tenant-Aware Queries

| Requirement | Test Must Verify |
|-------------|-----------------|
| List endpoints return only owned data | Listing products for Tenant A returns only Tenant A's products, even if Tenant B has products |
| Count endpoints are tenant-scoped | Product count for Tenant A reflects only Tenant A's products |
| Search results are tenant-scoped | Searching returns only the authenticated tenant's matching records |
| Aggregate queries are tenant-scoped | Sum/average operations reflect only the authenticated tenant's data |

### 7. Tenant-Aware Writes

| Requirement | Test Must Verify |
|-------------|-----------------|
| Write operations are tenant-attributed | Creating a product as Tenant A assigns organization_id = Tenant A |
| Cannot write to another tenant's resources | Tenant A cannot update a resource owned by Tenant B |
| Bulk writes are tenant-scoped | A batch create operation assigns all new records to the authenticated tenant |

### 8. Tenant-Aware Updates

| Requirement | Test Must Verify |
|-------------|-----------------|
| Can update own resources | Tenant A can update resources with org_id = Tenant A |
| Cannot update another tenant's resources | Attempting to update a resource with org_id = Tenant B fails (404) |
| Update does not change ownership | An update to a resource does not change its organization_id |

### 9. Tenant-Aware Deletion

| Requirement | Test Must Verify |
|-------------|-----------------|
| Can delete own resources | Tenant A can delete resources with org_id = Tenant A |
| Cannot delete another tenant's resources | Attempting to delete a resource with org_id = Tenant B fails (404) |
| Deletion does not affect other tenants | Deleting Tenant A's resource does not affect Tenant B's data |

### 10. Cache Isolation

| Requirement | Test Must Verify |
|-------------|-----------------|
| Cached data is tenant-scoped | A value cached for Tenant A is never returned to Tenant B |
| Cache invalidation is tenant-scoped | Invalidating Tenant A's cache does not invalidate Tenant B's |
| Cache keys include tenant context | The cache interface rejects attempts to construct keys without tenant scope |

### 11. Search Isolation

| Requirement | Test Must Verify |
|-------------|-----------------|
| Search results are tenant-scoped | A search by Tenant A returns only Tenant A's indexed documents |
| Indexing is tenant-attributed | Documents indexed for Tenant A are not visible to Tenant B |
| Search suggestions are tenant-scoped | Autocomplete for Tenant A uses only Tenant A's data |

### 12. Event Isolation

| Requirement | Test Must Verify |
|-------------|-----------------|
| Events include tenant context | Published events contain the publishing tenant's organization ID |
| Subscribers receive only their tenant's events | Tenant B's subscriber never receives Tenant A's events |
| Cross-tenant event delivery does not occur | An event published by Tenant A is never routed to Tenant B |

### 13. Queue Isolation

| Requirement | Test Must Verify |
|-------------|-----------------|
| Queue messages carry tenant context | Every queued message includes organization context |
| Processing is tenant-scoped | A message from Tenant A is processed within Tenant A's context |
| Failed messages from one tenant do not block another | A poisoned message from Tenant A does not delay Tenant B's processing |

### 14. Background Job Isolation

| Requirement | Test Must Verify |
|-------------|-----------------|
| Jobs carry tenant context | A background job triggered by Tenant A executes within Tenant A's scope |
| Jobs cannot access another tenant | A job for Tenant A cannot query Tenant B's data |
| Job failures are tenant-isolated | A failing job for Tenant A does not affect Tenant B's jobs |

### 15. File and Media Isolation

| Requirement | Test Must Verify |
|-------------|-----------------|
| Uploaded files are tenant-attributed | A file uploaded by Tenant A is owned by Tenant A |
| Files from another tenant are inaccessible | Tenant A cannot retrieve Tenant B's files (even with the correct path/URL) |
| File listing is tenant-scoped | Listing files returns only the authenticated tenant's assets |

### 16. Export Isolation

| Requirement | Test Must Verify |
|-------------|-----------------|
| Export contains only the authenticated tenant's data | An export for Tenant A includes no records from Tenant B |
| Export files are tenant-scoped | Generated export files are accessible only by the owning tenant |

### 17. Import Isolation

| Requirement | Test Must Verify |
|-------------|-----------------|
| Imported data is attributed to the authenticated tenant | Data imported by Tenant A is owned by Tenant A |
| Import cannot reference another tenant's resources | An import referencing Tenant B's IDs fails validation |

### 18. Webhook Isolation

| Requirement | Test Must Verify |
|-------------|-----------------|
| Outbound webhooks contain only the owning tenant's data | Webhook payloads for Tenant A do not include Tenant B's data |
| Inbound webhooks resolve to the correct tenant | An inbound webhook is processed within the registered tenant's context |
| Tenant A's webhook failure does not affect Tenant B | Delivery failure for Tenant A does not delay Tenant B's deliveries |

### 19. Notification Isolation

| Requirement | Test Must Verify |
|-------------|-----------------|
| Notifications are tenant-scoped | A notification triggered by Tenant A is delivered only to Tenant A's users |
| Notification content is tenant-scoped | Templates render with only the triggering tenant's data |

### 20. Configuration Isolation

| Requirement | Test Must Verify |
|-------------|-----------------|
| Configuration resolution is tenant-scoped | Tenant A's configuration values are not visible to Tenant B |
| Configuration changes are tenant-isolated | Changing a setting for Tenant A does not affect Tenant B's resolved value |

### 21. Audit Isolation

| Requirement | Test Must Verify |
|-------------|-----------------|
| Audit queries return only the authenticated tenant's records | Tenant A's audit API returns only Tenant A's audit entries |
| Audit entries from one tenant are not visible to another | Tenant B cannot see Tenant A's audit events |

### 22. Log Redaction

| Requirement | Test Must Verify |
|-------------|-----------------|
| Logs do not expose cross-tenant data | A log entry for Tenant A does not contain Tenant B's data |
| Log queries (if tenant-visible) are tenant-scoped | Any tenant-facing log access returns only their entries |

### 23. Support Impersonation

| Requirement | Test Must Verify |
|-------------|-----------------|
| Impersonation is scoped to one tenant | A support session targeting Tenant A cannot access Tenant B |
| Impersonation is time-limited | The session expires automatically |
| Impersonation is audited | The tenant's audit trail shows the impersonation event |
| Write access requires additional authorization | Read-only by default |

### 24. Tenant Suspension

| Requirement | Test Must Verify |
|-------------|-----------------|
| Suspended tenant cannot perform operations | API requests for a suspended tenant return 403 |
| Other tenants are unaffected | Suspension of Tenant A does not affect Tenant B |
| Data is preserved during suspension | After reactivation, data is intact |

### 25. Tenant Deletion

| Requirement | Test Must Verify |
|-------------|-----------------|
| Deleted tenant's data is gone | Queries for a deleted tenant return no results |
| Other tenants are unaffected | Deletion of Tenant A does not affect Tenant B's data |
| Deleted tenant's credentials are rejected | API keys and tokens for the deleted tenant are permanently rejected |

### 26. Tenant Restoration

| Requirement | Test Must Verify |
|-------------|-----------------|
| Restored tenant can operate normally | After restoration, Tenant A can perform business operations |
| Old credentials are not reconnected | Pre-deactivation API keys are permanently invalid. New keys are required. |
| Webhooks are not automatically restored | Webhook registrations must be re-created |

### 27. Cross-Tenant Administrative Operations

| Requirement | Test Must Verify |
|-------------|-----------------|
| Normal APIs do not permit cross-tenant access | No standard API endpoint returns data from another tenant |
| Support impersonation has defined boundaries | Support cannot escalate beyond their granted scope |
| Platform admin does not automatically access tenant data | Platform administration role alone does not return tenant business data |

## Reusable Negative-Test Pattern

Every isolation surface must implement this pattern:

```
1. Create Tenant A (organization, store, data).
2. Create Tenant B (organization, store, data).
3. Authenticate as a principal belonging ONLY to Tenant A.
4. Attempt to access Tenant B's resources (by known ID, by list, by search).
5. VERIFY: Request is denied (404 for resource access, empty for list/search).
6. VERIFY: No metadata about Tenant B's resources is leaked (count, existence, IDs).
7. VERIFY: No state change occurs in Tenant B (no writes, no deletions, no events).
8. VERIFY: Appropriate audit entry is generated (authorization failure logged).
9. VERIFY: Security monitoring signal is generated (cross-tenant access attempt).
```

### Pattern Rules

- This pattern is applied to every endpoint that accesses tenant-owned data.
- The pattern uses valid, existing resource IDs from Tenant B (not random IDs). This verifies that ID knowledge does not grant access.
- The pattern verifies the response body contains no information about Tenant B.
- The pattern verifies response timing does not reveal whether the resource exists (consistent response time for 404 regardless of existence).

## Test Categories

### Unit Tests

| Focus | Isolation Concern |
|-------|------------------|
| Data transfer objects | Cannot be constructed without tenant context |
| Validation rules | Tenant-scoped uniqueness constraints work correctly |
| Business logic | Calculations operate within tenant-scoped data only |

### Policy Tests

| Focus | Isolation Concern |
|-------|------------------|
| Permission evaluation | Permissions from one org do not apply in another |
| Membership validation | Non-members are denied regardless of permissions |
| Scope boundaries | Store-scoped permissions do not cross stores |

### Service Tests

| Focus | Isolation Concern |
|-------|------------------|
| Application service layer | Services receive and use tenant context correctly |
| Cross-module calls | Tenant context is preserved across module boundaries |
| Event publication | Published events include correct tenant context |

### Repository and Data-Access Tests

| Focus | Isolation Concern |
|-------|------------------|
| Query scoping | Every query includes tenant filter |
| Write attribution | Every insert includes correct tenant ownership |
| Unscoped query prevention | Attempting to query without tenant context is rejected |
| Cross-tenant join prevention | Queries cannot join data across organizations |

### Integration Tests

| Focus | Isolation Concern |
|-------|------------------|
| End-to-end request flow | Tenant context propagates from gateway through all layers |
| Multi-module operations | Checkout, order creation — all steps scoped correctly |
| Event-driven flows | Events trigger actions within the correct tenant |

### API Tests

| Focus | Isolation Concern |
|-------|------------------|
| Negative pattern (per endpoint) | Cross-tenant access denied for every endpoint |
| Response content | No cross-tenant data in responses |
| Error responses | 404 for out-of-scope (no information leakage) |
| Concurrent tenants | Parallel requests from different tenants resolve correctly |

### Event-Consumer Tests

| Focus | Isolation Concern |
|-------|------------------|
| Consumer receives only their tenant's events | No cross-tenant delivery |
| Consumer processes within tenant context | All downstream actions are tenant-scoped |
| Consumer failure isolation | One tenant's failed consumer doesn't block others |

### Background-Job Tests

| Focus | Isolation Concern |
|-------|------------------|
| Job carries tenant context | Context is preserved from trigger to execution |
| Job operates within context | All data access is scoped to the job's tenant |
| Job without context fails | Missing context results in job rejection, not wrong-tenant execution |

### End-to-End Tests

| Focus | Isolation Concern |
|-------|------------------|
| Complete business flow | Browse → Cart → Checkout → Order operates within one tenant |
| Parallel tenant flows | Two tenants executing flows simultaneously do not interfere |
| Cross-cutting services | Audit, notifications, events all scoped correctly during a full flow |

### Adversarial Tests

| Focus | Isolation Concern |
|-------|------------------|
| ID guessing/enumeration | Providing IDs from other tenants never succeeds |
| Header manipulation | Client-supplied tenant headers do not override authenticated context |
| Token manipulation | Modified tokens are rejected |
| Parameter injection | Query parameter attempts to access cross-tenant data are denied |
| Concurrent races | Race conditions do not create cross-tenant data exposure |

### Regression Tests

| Focus | Isolation Concern |
|-------|------------------|
| Previous isolation failures | Any fixed isolation bug has a permanent regression test |
| Schema changes | Migrations do not accidentally remove tenant scoping |
| Dependency updates | Library updates do not introduce tenant-scoping regressions |

### Load and Noisy-Neighbor Tests

| Focus | Isolation Concern |
|-------|------------------|
| High-volume tenant | One tenant's high load does not degrade another's response time beyond threshold |
| Resource exhaustion | One tenant approaching limits does not cause errors for other tenants |
| Rate limit isolation | One tenant's rate-limited requests do not reduce another's capacity |

### Backup and Restoration Tests

| Focus | Isolation Concern |
|-------|------------------|
| Backup attribution | Backup contains data attributed to correct tenants |
| Restoration scoping | Restoring one tenant's data does not inject into or overwrite another's |
| Post-restoration isolation | After restoration, tenant isolation holds as before |

## Module Onboarding Checklist

Every module that stores or processes tenant-owned resources must satisfy:

| # | Requirement | Evidence |
|---|-------------|---------|
| 1 | All entities include organization_id (non-nullable) | Schema review |
| 2 | All queries include tenant filter | Data-access tests |
| 3 | Negative-test pattern implemented for every endpoint | API tests |
| 4 | Cache keys include tenant prefix | Cache isolation tests |
| 5 | Events include tenant context in envelope | Event-consumer tests |
| 6 | Background jobs carry tenant context | Job isolation tests |
| 7 | Search results are tenant-scoped | Search isolation tests |
| 8 | Export/import operations are tenant-scoped | Export/import tests |
| 9 | Resource ownership verified on access | Policy + API tests |
| 10 | Cross-tenant access returns 404 (not 403) | API adversarial tests |
| 11 | No cross-tenant metadata leakage in responses | API adversarial tests |
| 12 | Audit events include tenant context | Integration tests |
| 13 | Test fixtures use independent tenants | Test review |
| 14 | Regression tests added for any isolation fix | Regression suite |

### Onboarding Rules

- This checklist is completed before a module's first release.
- Checklist compliance is reviewed during the module's design review and release approval.
- Partial completion does not pass. All items are required.
- **Every future module that stores or processes tenant-owned resources must satisfy this checklist.** No module is exempt.

## Release Gate

**Tenant-isolation failures block release.**

| Gate Condition | Outcome |
|---------------|---------|
| All tenant-isolation tests pass | Gate passes |
| Any tenant-isolation test fails | Gate blocks release |
| New endpoint without negative-test pattern | Gate blocks release |
| Module without complete onboarding checklist | Gate blocks release |
| Isolation regression introduced | Gate blocks release |
| Test coverage below defined threshold for tenant-aware surfaces | Gate blocks release |

### Gate Rules

- The gate is automated in the CI/CD pipeline. It cannot be bypassed.
- The gate applies equally to human-written and AI-generated code.
- A failing gate requires a fix, not a bypass or exception.
- The gate applies to every deployment, including hotfixes. Hotfixes that fix tenant isolation still verify all other isolation tests pass.
- Waiving the gate requires an ADR with security architect approval (exceptional, time-limited, with compensating controls).

## Decision Summary

| Decision | Rationale |
|----------|-----------|
| Negative testing is mandatory | Positive tests (own data accessible) do not prove isolation. Only negative tests (other tenant's data is inaccessible) prove isolation. |
| Two-tenant minimum for all isolation tests | Single-tenant tests cannot verify isolation. Two tenants are the minimum to verify that scoping works. |
| 404 for cross-tenant resources (not 403) | 403 reveals the resource exists. 404 reveals nothing. Tests verify the correct denial response. |
| Release gate is automated and non-bypassable | Manual approval is forgettable. Automated gates enforce consistently. |
| AI code follows same requirements | AI agents generate code quickly. Without the same testing requirements, they could introduce unverified isolation gaps at speed. |
| Every surface has dedicated tests | Isolation at the API level does not guarantee isolation at the cache, event, or job level. Each surface is tested independently. |
| Module onboarding checklist is mandatory | Without a checklist, new modules may ship without isolation tests. The checklist makes requirements explicit and verifiable. |

---

## Alternatives Considered

| Alternative | Rejected Because |
|------------|-----------------|
| Positive-only testing (verify own data accessible) | Does not prove isolation. Own-data tests pass even if cross-tenant access is also possible. |
| Single-tenant test fixtures | Cannot verify isolation. Two tenants are the minimum to demonstrate that scoping prevents cross-tenant access. |
| Optional isolation testing for "low-risk" modules | All modules that store tenant data are high-risk for isolation. No module is exempt. |
| Manual review instead of automated gate | Manual gates are skipped under pressure. Automated gates are consistent. |
| Isolation testing only at the API layer | Cache, event, search, and job isolation failures would go undetected. Each layer must be independently tested. |
| AI-generated code tested less rigorously | AI generates code faster, meaning isolation gaps propagate faster. Same rigor is more important, not less. |

---

## Trade-offs

| Trade-off | Accepted Because |
|-----------|-----------------|
| Multi-tenant fixtures increase test setup time | Test correctness is more important than test speed. Shared fixtures produce false confidence. |
| Negative-test pattern per endpoint increases test count | Each additional test prevents a specific cross-tenant vulnerability. The investment prevents catastrophic trust failures. |
| Automated gate may occasionally block valid deployments | False blocking is preferable to missed isolation failures reaching production. False blocks are investigated and resolved. |
| Testing every surface independently adds redundancy | Some isolation failures affect only one surface (cache but not API). Independent testing catches surface-specific failures. |
| Onboarding checklist adds friction to new modules | Friction at module creation prevents unfixable isolation gaps discovered in production. Early investment saves later incidents. |
